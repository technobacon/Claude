# 09 — Architecture & Decision Log (the "why")

*Last updated: 2026-06-05*

This is the narrative record of **how Forkful was built and why each decision was
made** — the reasoning, the trade-offs, and the dead-ends we deliberately
avoided. Read this to understand *why the code looks the way it does* before
changing it. (For "what to do next," see
[`07-handoff-and-next-steps.md`](07-handoff-and-next-steps.md).)

Format: each decision is **Context → Decision → Why → Trade-offs**.

---

## 0. The product, in one line

A Tinder-style recipe **discovery** app: swipe right to save, left to skip, tap
to cook on the **original publisher's site**. Long-term moat: a **pantry**
("what can I cook right now?"). Built web-first, mobile-tested on a real iPhone.

---

## 1. Where recipes come from — licensed APIs, not scraping

- **Context:** The brief said "pull from very well-regarded recipe sites,
  including pictures and reviews." The obvious-but-naive approach is to scrape
  those sites.
- **Decision:** **Option B — licensed recipe APIs** (TheMealDB, then Spoonacular),
  and **deep-link out** to the publisher for the full recipe. Never mirror
  photos/headnotes/reviews.
- **Why:**
  - **Legal.** A bare ingredient list isn't copyrightable, but **photos, the
    creative headnote prose, and reviews are**, and nearly every quality site's
    **Terms of Service forbid scraping**. The two things the brief most wanted to
    copy (pictures + reviews) are exactly the most legally dangerous.
  - **Maintenance.** Scrapers break constantly and fight bot protection.
  - **Posture.** Deep-linking out makes us a *discovery partner* that sends
    publishers traffic — which is also a future business model (affiliate /
    referral) rather than a lawsuit.
- **Trade-offs:** We depend on third-party APIs (quota, terms). Mitigated by the
  source-agnostic layer below and (later) caching.
- Detail: [`00-viability-analysis.md`](00-viability-analysis.md),
  [`05-legal-compliance.md`](05-legal-compliance.md).

## 2. A source-agnostic core (`RecipeSource` adapter)

- **Context:** "The more sources the better," and a desire to later add a
  structured-data/"publisher bridge" (Option C).
- **Decision:** Every source implements one `RecipeSource` interface
  (`search`/`getById`/`normalize`) and produces one **canonical `Recipe`**. The
  aggregator, API, and UI speak only the canonical shape.
- **Why:** Adding a source (or the Option C bridge) becomes *one file + one line*;
  one source failing only degrades variety, never breaks the feed; the pure
  `normalize()` is unit-testable with fixtures (no network).
- **Trade-offs:** An upfront abstraction before it was strictly needed — paid off
  immediately when Spoonacular slotted in beside TheMealDB.

## 3. Pure logic + Node's test runner, zero deps

- **Context:** The repo's existing ethos is "no build step, no runtime deps,
  testable pure functions."
- **Decision:** Keep all decision-making (normalize, dedupe, difficulty, deck
  ordering) in **pure functions** tested with `node --test`. Use **TypeScript via
  Node 22's native type-stripping** so tests need no install/build.
- **Why:** Fast, dependency-free tests; the logic that matters is verifiable in
  isolation. 23 tests guard the core.
- **Trade-offs:** `.ts` import extensions + type-stripping is slightly unusual,
  but Next + tsc tolerate it and it keeps `npm test` instant.

## 4. The pivotal one: a single-file **standalone** client for phone testing

- **Context:** The user wanted to test **on a real iPhone immediately**, and the
  Next.js app on Vercel was a repeated source of pain (monorepo Root Directory,
  branch confusion, mobile dashboard). The dev sandbox can't host anything the
  phone can reach.
- **Decision:** Build `standalone/index.html` — a **dependency-free, single-file**
  version that calls the recipe API **directly from the browser** and stores
  swipes/saves in **`localStorage`**. Serve it with **zero hosting** via a
  GitHub raw CDN (`raw.githack.com`), opened by a link pinned to a commit SHA.
- **Why:**
  - **It removed the hosting blocker entirely** — the user could *feel* the
    product on their phone in seconds, which unblocked rapid iteration (this is
    where most of the UX progress happened).
  - `localStorage` for state means **no server is required**, which later turned
    out to be exactly right for serverless (see §11).
  - SHA-pinned links are immutable and shareable; each change = a fresh link.
- **Trade-offs:** It **diverged** from the Next.js app's React UI (the standalone
  is ahead on UX). We accepted the divergence to keep momentum, keeping the
  *shared logic* (difficulty, spacing) in the tested TS core. Reconciling is a
  known future task (see §13, handoff).

## 5. PWA (Add to Home Screen)

- **Decision:** Manifest + Apple meta + icons (generated by a tiny pure-Node PNG
  encoder, matching the repo's "no image libs" convention).
- **Why:** A home-screen, full-screen launch is the natural bridge to "feels like
  an app" *before* investing in native — and costs almost nothing.

## 6. Real randomization (`random.php`), not alphabetical batches

- **Context:** The deck "felt not random."
- **Decision:** For the unfiltered deck, pull from the source's **whole-database
  random endpoint** (many calls), not "all meals starting with letter X."
- **Why:** First-letter batches are an *alphabetical slice*, not a random sample —
  that was the actual bug. True random sampling fixed the feel.
- **Trade-offs:** More requests per load; fine for testing, and a reason caching
  matters later.

## 7. Cuisine spacing in the deck

- **Decision:** After shuffling, reorder so the **same cuisine doesn't repeat
  within ~3 cards** (`spaceByKey`, greedy with relaxation).
- **Why:** Perceived variety; the user explicitly disliked seeing the same country
  back-to-back. Kept as tested pure logic shared by both clients.

## 8. Derived **difficulty** — and why the first version was wrong

- **Context:** No recipe API ships a reliable "difficulty" field. v1 used
  *ingredient count + sentence count* and mislabeled a toss salad as "hard."
- **Decision:** Rewrite to be **technique-first**: weight hard cooking techniques
  (braise, sear, fold, temper, emulsify…), count **real steps by line breaks**
  (not sentences), add a long-time signal, and pull toss/no-cook/assemble toward
  **easy**. When the source gives a real cook time (Spoonacular `readyInMinutes`),
  switch to **time-first** scoring.
- **Why:** Difficulty is about *technique and time*, not list length. A
  6-ingredient braise is harder than a 12-ingredient salad. Verified against the
  exact dishes that were mislabeled.
- **Trade-offs:** Still a heuristic on text — occasionally wrong on run-on
  paragraphs/unusual words. The honest fix is richer structured data
  (Spoonacular steps/equipment/time), which time-first scoring already leans on.

## 9. Filters: chips → dropdowns → **multi-select popovers**

- **Context:** Filters started as chips (space-hungry), the user asked for
  dropdowns (compact), then asked to **tick multiple** options.
- **Decision:** Each filter is a button that opens a **checklist popover**
  (multi-select); `maxTime` stays single-choice. Values are arrays; the data
  layer fans out as needed (e.g. one Spoonacular call per meal type since its
  `type` param is single-valued; cuisines joined as OR; diets as AND).
- **Why:** Reconciles "compact" with "select multiple," which native multi-select
  can't do well on mobile.
- **Trade-offs:** More client logic; meal-type multi-select costs extra API calls.

## 10. Adding Spoonacular — and the API-key dilemma

- **Context:** TheMealDB has no cook time, weak diets, no allergens. Spoonacular
  has all of it (`readyInMinutes`, diets, intolerances, nutrition).
- **Decision:** Add Spoonacular. **First** as a *bring-your-own-key* flow: the
  user pastes their key into a ⚙ Settings sheet, stored **only in their device's
  `localStorage`**.
- **Why:** A key in a **public** repo gets scraped by bots within minutes and the
  free quota drained. On-device storage keeps the key private and needed **zero
  hosting** to start using Spoonacular immediately.
- **Trade-offs:** Bring-your-own-key isn't viable for real end users — which
  forced the next decision.

## 11. Keyless for everyone — a **stateless serverless proxy** holding one key

- **Context:** "Can users avoid getting their own key?" A shared key **must** live
  on a server (a public file can't hide it).
- **Decision:** A tiny **dependency-free serverless function** (`/api/feed`,
  `/api/meta`) holds the key in a **server env var** (`SPOONACULAR_API_KEY`) and
  proxies Spoonacular. The standalone **auto-detects** this backend
  (`?api=<url>` or same-origin `/api/meta`) and runs **keyless**; personal state
  stays in the browser.
- **Why each part:**
  - **A proxy, not the full Next app:** avoids re-porting the (loved) standalone
    UI into React, and avoids the serverless **state problem** — Vercel functions
    are stateless/ephemeral, so server-memory swipe storage would be unreliable.
    Keeping personal state in `localStorage` makes the server purely a stateless
    recipe fetcher, which is the *correct* serverless design.
  - **Auto-detect via `/api/meta`:** one HTML file works **both** keyless (served
    by/pointed at a backend) and direct (free TheMealDB or personal key) — no
    separate builds.
  - **CORS open on the API:** the CDN-hosted client (different origin) can call
    the deployed server.
- **Trade-offs:** Quota is now **shared** across all users on one key — fine for
  testing; before real traffic, add server-side caching and/or a paid tier.

## 12. The deploy: dodging Vercel's monorepo trap

- **Context:** The user "never has luck with Vercel." The recurring failure was
  the **Root Directory** setting (the app lived in `recipe-swipe/`, a subfolder
  next to the old `tabstash/`), which is buried and painful on mobile. A 404 at
  the deploy root repeatedly confused things.
- **Decisions & why:**
  - **Dedicated repo was the goal** (app at root ⇒ no Root Directory). The GitHub
    integration **couldn't create a new repo** (403), so we pivoted.
  - **Put the proxy at the *repo root*** (`/api/*.js` + a root `package.json`).
    Vercel auto-serves root `/api/*` as functions for an "Other" project with
    **no Root Directory and no build step** — sidestepping the exact thing that
    kept failing.
  - **Merged to the default branch via PR** so Vercel's default import "just
    works" (no production-branch setting). A leftover `next` version conflict
    (`15.5.9` on default vs the security patch `15.5.19` on our branch) was
    resolved in our favor by merging the branches and keeping `15.5.19`.
  - **The root `/` 404 is expected** — the deploy is API-only; the client is the
    CDN-hosted standalone pointed at it with `?api=`.
- **Trade-offs:** Two backends now exist (this proxy *and* the Next app's
  `/api/feed`). The proxy is the live one; the Next app remains the richer,
  typed/tested reference and the path to a single-origin hosted client later.
- Detail: [`08-deploy-shared-key.md`](08-deploy-shared-key.md).

## 13. Known tensions to resolve later (so they're not surprises)

- **Standalone ↔ Next app divergence.** The standalone leads on UX; the Next app
  leads on typed/tested structure. Pick one as the production client (recommended:
  serve the standalone from the app for a single clean URL, or port its UX into
  React) — keep shared logic in the tested core either way.
- **Difficulty is heuristic.** Good, not perfect; improves with structured data.
- **Shared quota.** Add caching before real traffic.
- **No accounts yet.** Saved lists are per-device (localStorage). Cross-device
  sync needs accounts + a DB (Phase 3).
- **TheMealDB filters one dimension server-side**; combined Type+Cuisine is
  filtered client-side (Spoonacular handles multi natively).

## Timeline (one glance)

1. Plan + viability → **licensed APIs + deep-link-out**.
2. Source-agnostic core + TheMealDB adapter + tested pure logic.
3. Next.js MVP **+** the **standalone single-file client** for real-phone testing.
4. Iteration on the standalone: PWA, true randomization, cuisine spacing,
   undo + detail sheet, technique/time **difficulty**, dropdown → **multi-select**.
5. **Spoonacular** (bring-your-own-key) → cook time, diets, allergens.
6. **Keyless**: stateless serverless **proxy** at repo root + client auto-detect.
7. **Deployed** to Vercel (no Root Directory), merged to default, **cards load
   keyless**. ✅
